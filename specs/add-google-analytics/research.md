# Research: add-google-analytics

## Executive Summary

The site is a static GitHub Pages single-page app. `index.html` loads Google Analytics, and `main.js` renders all hash-routed pages from Markdown content. The prior GA setup measured only the shell page load; the site needs manual SPA page views and interaction events so Google Analytics can distinguish internal pages, links, buttons, search, and content engagement.

## Codebase Analysis

### Pages And Routes

| Route | Source | Notes |
|------|--------|-------|
| `#/` | `renderHomeView` | Home layout with about preview, topics, featured post, latest posts, projects, skills, races, AI links, and contact links |
| `#/about` | `renderAboutView` | About/profile page with contact links, topics, and work/project cards |
| `#/posts` | `renderPostsView` | Posts listing |
| `#/posts?tag=<tag>` | `renderPostsView` | Topic-filtered posts listing |
| `#/posts?q=<query>` | `renderPostsView` | Search-filtered posts listing |
| `#/posts/<slug>` | `renderPostDetailView` | Post detail page |
| `#/projects` | `renderProjectsView` | Projects listing |
| `#/projects/<slug>` | `renderProjectDetailView` | Project detail page |

### Links

| Surface | Link Type | Examples |
|---------|-----------|----------|
| Topbar | Internal navigation | Brand, About, Posts, Projects |
| Home about preview | Internal navigation | Read more |
| Topic chips | Internal filtered navigation | Profile topics and post tags |
| Hero | Internal content navigation | Featured post media/title/shell |
| Cards | Internal content navigation | Post cards and project cards |
| Detail pages | Internal navigation | Back to posts, back to projects |
| Skill links | Resource or outbound links | Repository, docs |
| AI panel | Local resource links | `generated/claude-code-skill.md`, `generated/ai-context.json`, `llms.txt` |
| Contact lists | Contact/outbound links | Email, phone, website, LinkedIn |
| Markdown article body | Generic content links | Links generated by `inlineMarkdown` |

### Buttons And Interactions

| Surface | Interaction | Tracking Need |
|---------|-------------|---------------|
| Search form | Submit query and navigate to `#/posts?q=<query>` | Track query term and resulting route page view |
| Theme toggle | Toggle light/dark theme | Track theme selected |
| Hero shell | Click non-link area to navigate | Track featured post click |
| Hash route changes | SPA page change | Track manual `page_view` with route and content metadata |

## Constraints

- Keep measurement ID `G-EHDL9M60FS`.
- Avoid duplicate page views from GA's automatic page-load behavior and the app's manual hash route views.
- Do not add dependencies.
- Existing rendering, hash routing, search, and theme behavior must remain intact.

## Recommendations

1. Configure GA with `send_page_view: false` in `index.html`.
2. Add app-level helpers in `main.js` for manual `page_view` events and interaction events.
3. Attach `data-analytics-*` metadata to generated links and controls for stable event parameters.
4. Track standard and readable event names: `page_view`, `nav_click`, `post_card_click`, `project_card_click`, `topic_click`, `search`, `theme_change`, `click`, and specific resource/contact actions.
5. Include GA parameters that are useful in reports: `page_path`, `page_title`, `route_name`, `content_type`, `content_id`, `ui_region`, `link_url`, `link_text`, `link_domain`, `outbound`, and `search_term`.

## Open Questions

- None for implementation. In GA4, custom parameters such as `route_name`, `content_type`, `content_id`, `ui_region`, and `link_action` may need to be registered as custom dimensions before they are convenient in standard reports.
